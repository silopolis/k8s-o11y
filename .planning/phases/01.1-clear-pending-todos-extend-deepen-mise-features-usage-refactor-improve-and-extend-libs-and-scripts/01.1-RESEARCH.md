## 1. mise Secrets with SOPS

### 1.1 Overview

mise provides built-in secrets management via SOPS (Secrets OPerationS) with age encryption. This is the **recommended approach** for this project.

**Key Capabilities:**
- Encrypt `.env.json`, `.env.yaml`, or `.env.toml` files
- Automatic decryption when `mise env` runs
- Environment variables sourced from encrypted files
- Redaction support for CI/CD masking

### 1.2 Configuration

**File Structure:**
```toml
# mise.toml
[env]
_.file = ".env.json"  # Automatically decrypted if SOPS-encrypted
```

**Setup Workflow:**
```bash
# 1. Install tools
mise use -g sops age

# 2. Generate age key
age-keygen -o ~/.config/mise/age.txt
# Note the public key output

# 3. Create secrets file
echo '{"GRAFANA_ADMIN_PASSWORD": "secret123"}' > .env.json

# 4. Encrypt with sops
sops encrypt -i --age "<public_key>" .env.json

# 5. Reference in mise.toml
```

### 1.3 Environment Variables

**Priority Order (highest to lowest):**
1. `MISE_SOPS_AGE_KEY` - Direct key content
2. `MISE_SOPS_AGE_KEY_FILE` - Path to key file
3. `SOPS_AGE_KEY_FILE` - Standard SOPS path
4. `SOPS_AGE_KEY` - Standard SOPS direct key
5. Default: `~/.config/mise/age.txt`

### 1.4 Integration Points

**With Helmfile:**
```yaml
# helmfile.yaml
releases:
  - name: kube-prometheus-stack
    values:
      - values/kube-prometheus-stack.yaml
      - values/secrets.yaml  # References env vars
```

**With CI/CD:**
```yaml
# .github/workflows/deploy.yaml
- name: Mask secrets
  run: |
    for value in $(mise env --redacted --values); do
      echo "::add-mask::$value"
    done
```

### 1.5 Best Practices

1. **Commit encrypted files** - Safe to version control
2. **Never commit `.env` plaintext** - Add to `.gitignore`
3. **Use redaction** for sensitive values in CI
4. **Document key distribution** - How team members get age keys
5. **Separate concerns** - Infrastructure secrets vs. personal API keys

**Confidence:** HIGH (official mise documentation, tested pattern)

## 2. Pre-Commit Framework

### 2.1 Overview

pre-commit is a framework for managing git hooks that run checks before commits. It supports multi-language hooks and is the industry standard for pre-commit checks.

**Key Benefits:**
- Runs only on staged files (fast)
- Supports auto-fix for many issues
- Version-controlled configuration
- Easy team sharing

### 2.2 Installation

**As mise tool:**
```toml
# mise.toml
[tools]
pre-commit = "latest"
```

**Install hooks:**
```bash
# After adding .pre-commit-config.yaml
pre-commit install
```

### 2.3 Configuration

**Basic `.pre-commit-config.yaml`:**
```yaml
# .pre-commit-config.yaml
default_install_hook_types: [pre-commit, commit-msg]

repos:
  # Built-in hooks
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace
        args: [--markdown-linebreak-ext=md]
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-json
      - id: check-added-large-files
        args: [--maxkb=1000]

  # Markdown linting
  - repo: https://github.com/DavidAnson/markdownlint-cli2
    rev: v0.17.0
    hooks:
      - id: markdownlint-cli2
        args: [--fix]

  # Shellcheck
  - repo: https://github.com/shellcheck-py/shellcheck-py
    rev: v0.10.0.1
    hooks:
      - id: shellcheck
        args: [-x, -e, SC2034]
```

### 2.4 Hook Behavior

**Auto-fix then fail:**
- Some hooks (`trailing-whitespace`, `end-of-file-fixer`, `markdownlint-cli2 --fix`) modify files
- If files were modified, commit fails
- User must re-stage and re-commit
- This ensures clean commits

**Fail only (no auto-fix):**
- `check-yaml`, `check-json`, `shellcheck`
- Must be manually fixed

### 2.5 Staged Files Only

By default, pre-commit only checks files being committed:
```bash
# Check staged files only (default)
pre-commit run

# Check all files (CI use)
pre-commit run --all-files
```

**Confidence:** HIGH (official pre-commit docs, widely used)

## 4. Shellcheck Configuration

### 4.1 Overview

Shellcheck is the industry-standard static analysis tool for shell scripts. It catches:
- Unquoted variables (word splitting)
- Deprecated syntax
- Unused variables
- Missing error handling
- Portability issues

### 4.2 mise Integration

**As mise tool:**
```toml
# mise.toml
[tools]
shellcheck = "latest"
```

**Task definition:**
```toml
[tasks.lint_shell]
description = "Lint shell scripts"
run = "shellcheck scripts/*.sh lib/*.sh"

[tasks.lint_shell_strict]
description = "Lint with warnings as errors"
run = "shellcheck -S warning scripts/*.sh lib/*.sh"
```

### 4.3 Configuration

**Via `.shellcheckrc`:**
```bash
# .shellcheckrc
# Shell dialect
shell=bash

# Disable specific checks
disable=SC2034  # Unused variables (common in library files)

# Enable optional checks
enable=add-default-case
enable=check-set-e-suppressed
enable=deprecate-which

# Source path for finding sourced files
source-path=lib
source-path=scripts
```

### 4.4 Severity Levels

| Level | Code | Behavior |
|-------|------|----------|
| error | -S error | Critical issues only |
| warning | -S warning | Include likely bugs |
| info | -S info | Include style issues |
| style | -S style | Everything (default) |

**Recommendation:** Start with `-S warning` for pre-commit, `-S style` for manual runs.

### 4.5 Library Script Handling

Library files (like `lib/*.sh`) often trigger `SC2034` (unused variables) because:
- Functions define variables for callers to use
- Variables are used by sourcing scripts

**Solutions:**
1. Disable SC2034 in `.shellcheckrc` globally
2. Use `export` for variables meant to be used externally
3. Add `# shellcheck disable=SC2034` to specific lines

**Confidence:** HIGH (established tool, clear documentation)

## 5. Mise Tasks Directory Conventions

### 5.1 Overview

mise supports **file tasks** - standalone executable scripts in specific directories. This is the modern replacement for inline `[tasks]` in `mise.toml`.

**Benefits:**
- Syntax highlighting and linting
- Easier to edit and test
- Self-documenting structure
- Version control friendly

### 5.2 Directory Options

mise discovers tasks in these directories (in order):
1. `.config/mise/tasks/`
2. `.mise/tasks/` ⭐ **Recommended for this project**
3. `mise/tasks/`
4. `.mise-tasks/`
5. `mise-tasks/`

### 5.3 File Structure

**Hierarchical naming:**
```
.mise/tasks/
├── check/
│   ├── preflight          # -> mise run check:preflight
│   └── versions           # -> mise run check:versions
├── deploy/
│   ├── crds               # -> mise run deploy:crds
│   └── stack              # -> mise run deploy:stack
├── lint/
│   ├── helmfile           # -> mise run lint:helmfile
│   ├── markdown           # -> mise run lint:markdown
│   └── shell              # -> mise run lint:shell
├── verify/
│   ├── phase1             # -> mise run verify:phase1
│   └── stack              # -> mise run verify:stack
└── utils/
    └── get-versions       # -> mise run utils:get-versions
```

### 5.4 Task File Format

**Basic structure:**
```bash
#!/usr/bin/env bash
#MISE description="Run preflight checks"
#MISE alias="preflight"

set -euo pipefail

# Script content here
```

**With dependencies:**
```bash
#!/usr/bin/env bash
#MISE description="Deploy full stack"
#MISE depends=["lint:helmfile", "deploy:crds"]

set -euo pipefail

# Script content here
```

**With arguments:**
```bash
#!/usr/bin/env bash
#MISE description="Verify specific component"
#USAGE arg "<component>" help="Component to verify"

set -euo pipefail

echo "Verifying: ${usage_component}"
```

### 5.5 Migration Strategy

**Hybrid approach (recommended):**
```toml
# mise.toml
# Keep simple one-liners inline
[tasks.cls_up]
description = "Create Talos cluster"
run = "talosctl cluster create docker --name ..."
quiet = true

# Move complex tasks to files
# .mise/tasks/deploy/stack will replace deploy_stack
```

**Migration steps:**
1. Create `.mise/tasks/` directory structure
2. Move complex tasks to files (one at a time)
3. Make files executable: `chmod +x .mise/tasks/deploy/stack`
4. Test each migrated task
5. Remove old inline definitions once migrated

### 5.6 Naming Conventions

- Use kebab-case for file names: `verify-phase1` not `verify_phase1`
- Use `_default` for directory default tasks
- Group by function: `deploy/`, `lint/`, `verify/`, `utils/`

**Confidence:** HIGH (official mise docs, recent feature)

## 6. Lib/ Directory Patterns for Bash

### 6.1 Overview

The project already has a `lib/` directory with shared libraries. This section documents best practices for extending it.

**Current structure:**
```
lib/
├── checks.sh      # Validation functions
├── output.sh      # Printing/color functions
└── talos.sh       # Talos-specific helpers
```

### 6.2 Library Patterns

**Naming conventions:**
- Files: `lib/<domain>.sh`
- Functions: `<domain>_<action>()` or `get_<resource>_<attribute>()`
- All functions use `snake_case`

**Library template:**
```bash
#!/usr/bin/env bash
#
# lib/kubernetes.sh
# Shared Kubernetes utility functions
#
# Usage: source "$(dirname "$0")/../lib/kubernetes.sh"
# Dependencies: lib/output.sh must be sourced first

# Get Kubernetes server version
# Returns: Version string (e.g., "v1.28.0") or empty on failure
get_kubernetes_version() {
    kubectl version -o json 2>/dev/null | \
        jq -r '.serverVersion.gitVersion' 2>/dev/null || echo ""
}

# Get node count by type
# Args: $1 = node type (control-plane, worker, or all)
# Returns: Count as integer
get_node_count() {
    local type="${1:-all}"
    local selector=""
    
    case "$type" in
        control-plane) selector="node-role.kubernetes.io/control-plane" ;;
        worker) selector="!node-role.kubernetes.io/control-plane" ;;
        *) selector="" ;;
    esac
    
    if [ -n "$selector" ]; then
        kubectl get nodes -l "$selector" --no-headers 2>/dev/null | wc -l
    else
        kubectl get nodes --no-headers 2>/dev/null | wc -l
    fi
}
```

### 6.3 Sourcing Pattern

**In scripts:**
```bash
#!/usr/bin/env bash
set -euo pipefail

# Calculate project root
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
PROJECT_ROOT="$(dirname "$SCRIPT_DIR")"

# Source libraries (order matters if there are dependencies)
source "${PROJECT_ROOT}/lib/output.sh"
source "${PROJECT_ROOT}/lib/checks.sh"
source "${PROJECT_ROOT}/lib/kubernetes.sh"
```

### 6.4 Getter Functions Pattern

**Design principles:**
1. Return raw values only (no formatting, no echo with colors)
2. Return empty string on failure (not error messages)
3. Use `|| true` or `|| echo ""` for error handling
4. Document return values in comments

**Example:**
```bash
# Get Prometheus CRD count
# Returns: Integer count, 0 if none or error
get_prometheus_crd_count() {
    kubectl get crd 2>/dev/null | \
        grep -c "monitoring.coreos.com" 2>/dev/null || echo "0"
}

# Get list of Prometheus CRDs
# Returns: Space-separated list of CRD names
get_prometheus_crds() {
    kubectl get crd 2>/dev/null | \
        grep "monitoring.coreos.com" | \
        awk '{print $1}' | \
        tr '\n' ' ' || echo ""
}
```

### 6.5 Testing Considerations

Since these are bash functions, traditional unit testing is hard. Alternatives:
1. **Manual testing** via test scripts
2. **Integration testing** in real cluster
3. **Shellcheck** for syntax validation
4. **Documentation** showing expected inputs/outputs

### 6.6 New Library: kubernetes.sh

**Proposed structure for TODO #10:**
```bash
# lib/kubernetes.sh - Getter functions for cluster/tool versions

get_kubectl_version()         # Client version
get_kubernetes_version()      # Server version
get_helm_version()              # Helm version
get_helmfile_version()          # Helmfile version
get_node_count()                # Total nodes
get_control_plane_node_count()  # Control plane nodes
get_worker_node_count()         # Worker nodes
get_nodes_not_ready()           # Count of not-ready
get_prometheus_crd_count()      # Number of monitoring CRDs
get_prometheus_crds()           # List of CRD names
```

## 7. Talos Control Plane Metrics

### 7.1 The Problem

Talos Linux binds control plane metrics to `127.0.0.1` (localhost only) for security. Prometheus can't scrape them.

**Default bind addresses:**
| Component | Default | Port | Needs Change |
|-----------|---------|------|--------------|
| kube-controller-manager | 127.0.0.1 | 10257 | Yes |
| kube-scheduler | 127.0.0.1 | 10259 | Yes |
| kube-proxy | 127.0.0.1 | 10249 | Yes |
| etcd | 127.0.0.1 | 2381 | Yes (separate process) |

### 7.2 Configuration Fix

**Patch file approach (already documented):**
```yaml
# config/talos/control-plane-metrics-patch.yaml
cluster:
  controllerManager:
    extraArgs:
      bind-address: 0.0.0.0
  scheduler:
    extraArgs:
      bind-address: 0.0.0.0
  proxy:
    extraArgs:
      metrics-bind-address: 0.0.0.0:10249
  etcd:
    extraArgs:
      listen-metrics-urls: http://0.0.0.0:2381
```

**Applying to nodes:**
```bash
# Rolling update across control plane nodes
talosctl -n <node-ip> apply-config -f <config> --insecure
```

### 7.3 Security Considerations

**Risk:** Binding to `0.0.0.0` exposes metrics on all interfaces.

**Mitigation:**
1. **Network policies** - Restrict Prometheus pod access only
2. **Firewall rules** - Limit to cluster network
3. **Documentation** - Record the configuration change for audits

### 7.4 Prometheus Scraping Configuration

**Disable default ServiceMonitors (kube-prometheus-stack values):**
```yaml
kubeControllerManager:
  enabled: false  # Disable default

kubeScheduler:
  enabled: false  # Disable default

kubeProxy:
  enabled: false  # Disable default

kubeEtcd:
  enabled: false  # Disable default
```

**Add custom scrape configs:**
```yaml
prometheus:
  prometheusSpec:
    additionalScrapeConfigs:
      # Controller Manager
      - job_name: 'talos-controller-manager'
        kubernetes_sd_configs:
          - role: node
        relabel_configs:
          - source_labels: [__address__]
            regex: ([^:]+):\d+
            replacement: ${1}:10257
            target_label: __address__
        scheme: https
        tls_config:
          insecure_skip_verify: true
        bearer_token_file: /var/run/secrets/kubernetes.io/serviceaccount/token
```

### 7.5 Verification

**Test metrics endpoint:**
```bash
# From a pod with hostNetwork
kubectl run test --rm -it --image=curlimages/curl --restart=Never \
  --overrides='{"spec":{"hostNetwork":true}}' -- \
  curl -s http://<node-ip>:10257/metrics | head
```

**Check Prometheus targets:**
```bash
kubectl port-forward svc/kube-prometheus-stack-prometheus 9090:9090 -n monitoring
# Open http://localhost:9090/targets
```

**Confidence:** MEDIUM-HIGH (Talos docs, community blog posts, but needs testing in actual cluster)

## 8. Integration Patterns

### 8.1 mise + Pre-Commit Integration

**Key insight:** Pre-commit hooks should use mise-managed tools.

**Pattern:**
```yaml
# .pre-commit-config.yaml
- repo: local
  hooks:
    - id: shellcheck
      name: Shellcheck
      entry: mise exec -- shellcheck
      language: system
      files: \.(sh|bash)$
```

**Alternative:** Let pre-commit install its own copies (simpler, but versions may drift).

### 8.2 mise + SOPS Integration

**Pattern for secrets:**
```toml
# mise.toml
[env]
_.file = ".env.json"  # Encrypted via SOPS

[secrets]
# Optional: configure SOPS settings
sops.age_key_file = "~/.config/mise/age.txt"
```

**Git workflow:**
1. Team member gets age private key out-of-band
2. Places it in `~/.config/mise/age.txt`
3. `mise env` automatically decrypts `.env.json`
4. Helmfile reads env vars as values

### 8.3 Quality Gate Sequence

**Recommended execution order:**

1. **Install phase:**
   ```bash
   mise install              # Install all tools
   pre-commit install        # Install git hooks
   ```

2. **Development phase:**
   ```bash
   git add <files>
   git commit              # Pre-commit hooks run automatically
   ```

3. **Pre-deployment phase:**
   ```bash
   mise run check:preflight   # Manual validation
   mise run lint:all          # Full linting
   mise run deploy:stack      # Deployment
   ```

### 8.4 Task Dependencies

**Example dependency chain:**
```
deploy:stack
  ├── lint:all
  │   ├── lint:markdown
  │   ├── lint:shell
  │   └── lint:helmfile
  ├── deploy:crds
  └── verify:stack
```

**Confidence:** HIGH (patterns derived from mise docs and pre-commit docs)

## 9. Common Pitfalls

### 9.1 mise Secrets

| Pitfall | Problem | Solution |
|---------|---------|----------|
| Uncommitted age key | New team member can't decrypt | Document key distribution |
| Wrong precedence | MISE_SOPS_AGE_KEY_FILE vs SOPS_AGE_KEY_FILE | Understand priority order |
| Forgetting `-i` flag | Creates new file instead of overwriting | Use `sops encrypt -i` |
| Committing plaintext | Secrets in git | Add `.env*.json` to `.gitignore` before encrypting |

### 9.2 Pre-Commit

| Pitfall | Problem | Solution |
|---------|---------|----------|
| Hooks not installed | pre-commit doesn't run | Run `pre-commit install` |
| Version drift | pre-commit uses different tool versions | Pin versions in config |
| Auto-fix confusion | Files modified but commit fails | Re-stage and re-commit |
| Slow hooks | Large repos check all files | Ensure `staged-only` behavior |

### 9.3 Shellcheck

| Pitfall | Problem | Solution |
|---------|---------|----------|
| SC2034 noise | Library variables "unused" | Disable in `.shellcheckrc` |
| Missing source-path | Can't find sourced files | Set `source-path` in config |
| Different shells | Syntax errors for bashisms | Set `shell=bash` explicitly |

### 9.4 Talos Metrics

| Pitfall | Problem | Solution |
|---------|---------|----------|
| Binding doesn't work | extraArgs syntax wrong | Verify Talos version compatibility |
| Metrics still unreachable | Firewall/network policy | Check node connectivity |
| etcd still failing | mTLS requirement | Extract etcd certs for Prometheus |
| Security exposure | 0.0.0.0 too broad | Add network policies |

### 9.5 mise Tasks Directory

| Pitfall | Problem | Solution |
|---------|---------|----------|
| Not executable | Task not discovered | `chmod +x .mise/tasks/<task>` |
| Wrong comment format | `# MISE` vs `#MISE` | No space after `#` |
| Formatter breaks config | `#MISE` → `# MISE` | Use `# [MISE]` alternative |
| Directory not recognized | Using wrong path | Check mise discovers it: `mise tasks` |

**Confidence:** HIGH (documented issues, community experience)

### 11.1 New Findings Since Initial Research

#### mise 2026.3.0 Release (March 2026)
Recent mise releases (v2026.3.0+) introduce significant improvements:

1. **`mise prepare` Overhaul:** Content-hash freshness detection (blake3), dependency ordering between providers, and `--explain` flag for diagnostics. This could be used for Wave 1 verification instead of manual checks.

2. **Task-Backed Hooks:** Hooks and `watch_files` can now reference full mise tasks with `task = "name"` syntax, gaining access to dependencies, env vars, and templating.

3. **Native Git Hook Generation:** `mise generate git-pre-commit` can now create git hooks that run mise tasks directly, offering an alternative to pre-commit framework.

#### hk: The mise-Native Git Hook Manager

**Major Discovery:** The mise project itself uses **hk** (not pre-commit) for its git hooks. hk is a Rust-based git hook manager by the same author (jdx) with:

- **Parallel execution** with file locking to prevent race conditions
- **Stash unstaged changes** before running fix hooks (prevents common pre-commit issue)
- **Built-in linters** for common tools (prettier, black, etc.)
- **Native mise integration** via `hk install --mise` flag

**Implication:** For this project, we have THREE options for quality gates:

| Approach | Pros | Cons | Recommendation |
|----------|------|------|----------------|
| **pre-commit** | Industry standard, hundreds of hooks, well-known | Slower, Python-based, requires separate install | Stick with this (team familiarity) |
| **hk** | Fast, Rust-native, perfect mise integration, file locking | Newer, smaller ecosystem, Pkl config | Consider for future migration |
| **mise generate git-pre-commit** | Simple, no extra dependencies | Basic functionality only | Not suitable for complex needs |

**Decision:** Keep pre-commit for Phase 01.1 (team knows it), but document hk as future consideration.

### 11.2 Better Integration Patterns

#### Pattern A: mise-managed pre-commit Tools (Recommended)

Instead of letting pre-commit install its own copies of tools (which may drift from mise versions):

```yaml
# .pre-commit-config.yaml
- repo: local
  hooks:
    - id: shellcheck
      name: Shellcheck
      entry: mise exec -- shellcheck
      language: system
      files: \.(sh|bash)$
    
    - id: markdownlint
      name: Markdownlint
      entry: mise exec -- markdownlint-cli2
      language: system
      files: \.md$
```

**Benefits:**
- Single source of truth for tool versions (mise.toml)
- Faster hook installation (no redundant downloads)
- Consistent versions between `mise run lint:shell` and pre-commit

### 11.4 Team Onboarding: Age Key Distribution

**Critical Gap Identified:** The initial research didn't address HOW team members get the age private key. This is a significant friction point.

**Recommended Approaches (in order of security):**

2. **Out-of-Band Communication:**
   - Slack DM with encrypted message
   - Email with encrypted attachment
   - In-person/Zoom screen share for initial setup

3. **Key Rotation Protocol:**
   ```bash
   # When a team member leaves
   # 1. Generate new age key
   age-keygen -o ~/.config/mise/age.txt.new
   
   # 2. Re-encrypt secrets with new key
   sops encrypt -i --age "<new_public_key>" .env.json
   
   # 3. Distribute new key to remaining team
   # 4. Old key can no longer decrypt
   ```

### 11.7 Common Pitfalls (New Discoveries)

#### Pitfall 1: mise SOPS Precedence Confusion

The precedence order changed in recent mise versions. **Current order (highest to lowest):**

1. `MISE_SOPS_AGE_KEY` (highest priority)
2. `MISE_SOPS_AGE_KEY_FILE` or `sops.age_key_file` setting
3. `SOPS_AGE_KEY_FILE` (standard SOPS)
4. `SOPS_AGE_KEY` (standard SOPS, direct key)
5. `~/.config/mise/age.txt` (default)

**Issue:** If someone sets `SOPS_AGE_KEY_FILE` expecting it to work, but also has an old key in `~/.config/mise/age.txt`, mise uses the default file instead of the env var.

**Fix:** Always use `MISE_SOPS_AGE_KEY_FILE` for mise-specific configuration.

#### Pitfall 2: Formatter Destroys #MISE Comments

Many formatters (shfmt, prettier for markdown) will change:
```bash
#MISE description="..."
```
to:
```bash
# MISE description="..."
```

**mise ignores `# MISE`** (with space) by design to avoid accidental configuration.

**Fix:** Use alternative syntax: `# [MISE] description="..."` which is formatter-safe.

#### Pitfall 3: pre-commit Auto-Fix Requires Re-Staging

When hooks modify files (trailing-whitespace, end-of-file-fixer, markdownlint --fix):
1. Hook modifies file
2. pre-commit returns non-zero exit
3. Commit fails
4. User must `git add` the modified files again
5. Re-commit

**Common confusion:** "Why did my commit fail? The hook fixed it!"

**Fix:** Document this clearly in AGENTS.md and README.md:
```markdown
## Pre-Commit Auto-Fix

If pre-commit modifies files during a commit:
1. Review the changes: `git diff --cached`
2. Re-stage: `git add <modified-files>`
3. Re-commit: `git commit -m "..."`
```

#### Pitfall 4: Task File Not Executable

File tasks **must** be executable or mise won't discover them.

**Symptom:** Task shows in `mise.toml` but not in `mise tasks` output.

**Fix:** 
```bash
chmod +x .mise/tasks/deploy/stack
```

**Recommendation:** Add a pre-commit hook to check for this:
```yaml
- repo: https://github.com/pre-commit/pre-commit-hooks
  hooks:
    - id: check-executables-have-shebangs
```

### 11.8 Risk Mitigation Strategies

#### Risk 1: SOPS Key Loss

**Scenario:** Only one team member has the age key, they leave/lose it.

**Mitigation:**
- Store key in team password manager from day one
- Document key rotation procedure
- Consider multiple age recipients (if supported by SOPS version)

#### Risk 2: pre-commit Slows Development

**Scenario:** Developers skip hooks because they're too slow.

**Mitigation:**
- Use `mise exec` to ensure fast tool startup
- Keep hooks focused (don't run full test suite)
- Document `git commit --no-verify` for emergencies (with warnings)

#### Risk 3: Wave 1 Breaks Existing Scripts

**Scenario:** Removing PATH assumptions breaks someone's workflow.

**Mitigation:**
- Keep PATH injection as fallback (see 11.6)
- Announce change in team chat/email
- Provide `mise doctor` task to verify setup

#### Risk 4: etcd Metrics Require Certificates

**Scenario:** Unlike controller-manager/scheduler, etcd uses mTLS.

**Mitigation:**
- Research etcd cert extraction from Talos (may require Talos API)
- Document if etcd metrics are Phase 01.1 or Phase 01.2 scope
- Fallback: document that etcd metrics require additional setup

### 11.9 Integration Tips

#### Tip 1: mise + pre-commit + SOPS Synergy

Create a `mise run setup` task that does everything:

```bash
#!/usr/bin/env bash

mise install

    echo "   Get the key from the team password manager and place it there."
    echo "   Then re-run: mise run setup"
    exit 1
fi

# 3. Verify SOPS working
echo "Testing SOPS decryption..."
if mise env | grep -q "GRAFANA_ADMIN_PASSWORD"; then
    echo "✓ SOPS working"
else
    echo "⚠️  SOPS not working. Check age key."
    exit 1
fi

# 4. Install pre-commit
echo "Installing pre-commit hooks..."
pre-commit install
pre-commit autoupdate

# 5. Run initial lint
echo "Running initial lint pass..."
pre-commit run --all-files || true  # May have fixes to apply

echo "=== Setup complete ==="
echo "Run 'mise run check:preflight' to verify cluster access"
```
